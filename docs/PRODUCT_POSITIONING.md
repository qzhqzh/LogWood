# 空心树洞产品定位

> 状态：Accepted / 产品定义的单一权威来源（SSOT）  
> 生效日期：2026-07-23  
> 对应 Epic：#15

## 1. 品牌定义

- 产品名称：**空心树洞**
- 仓库名称：`LogWood`
- 品牌副标题：**大浪淘沙，找寻灵感**
- 功能定位：**AI 灵感炼成与实践沉淀社区**

一句话说明：

> 从一个念头、一条链接或一次踩坑出发，经过试用、评测与反复打磨，把值得留下的经验炼成可复用、可验证、可快速上手的模板、提示词、工作流和技能包；同时保存这段路上的吐槽、失败、讨论、技术小结与深度反思。

“空心”代表留出空间，不被已有结论占满；“树洞”既承接随手发现和真实情绪，也保存经验成长的年轮。

“大浪淘沙”不等于按热度排名。平台通过真实使用、上下文、证据、版本、复现性和持续维护，把真正值得留下的内容逐步淘出来。

## 2. 产品边界

空心树洞不是：

- 单纯的 Prompt 搬运或收藏站
- 只看五星均分的模型、软件排行榜
- 泛 AI 新闻门户
- 没有关联对象的通用匿名聊天室
- 用 AI 批量生成内容、替代真实测试的内容农场

空心树洞要解决的问题是：

1. 好灵感和好资源散落在链接、聊天、截图、仓库和临时笔记中，难以继续利用。
2. 模型、软件、Prompt 和工作流的评价通常缺少版本、场景和证据，结论很快失效。
3. 试错过程中的失败、吐槽和判断变化没有被系统保存，最终只剩一个脱离上下文的“成品”。
4. 从发现到可复用资产之间缺少清晰的成熟度、评测协议和打包流程。

## 3. 核心模型：一条生命线，两条并行轨道

所有内容围绕同一个 Subject 持续生长，不因栏目变化反复复制对象。

### 3.1 资产进化线

```text
随手灵感 / 外部资源
  → 灵感池（待观察）
  → 试用与任务验证
  → 结构化评测 / 对比 / 淘汰
  → 模板或 Prompt 草案
  → 可复用 Workflow / Skill
  → 带示例、版本与 Quick Start 的技能包
  → 持续验证、升级或归档
```

建议的成熟度状态：

| 状态 | 含义 | 最小要求 |
|---|---|---|
| `inspiration` | 刚发现的念头或链接 | 标题或 URL、发现理由 |
| `candidate` | 值得继续观察 | 基本元数据、来源、初步分类 |
| `trying` | 正在试用或设计测试 | 任务、环境或试用计划 |
| `verified` | 已有一次可解释验证 | 上下文、结果、证据、测试日期 |
| `reusable` | 已整理成可复用资产 | 目标、输入、步骤、示例、边界 |
| `packaged` | 可快速上手的技能包 | Quick Start、版本、依赖、变更记录 |
| `archived` | 过时、失败或不再维护 | 归档原因、替代项或最后有效版本 |

成熟度描述内容被验证和整理的程度，不直接代表好坏。一个失败案例也可以是高价值的 `verified` 内容。

### 3.2 经验沉淀线

```text
即时吐槽 / 快评
  → 提问、讨论与求证
  → 实验记录与失败样本
  → 技术小结
  → 评测报告 / 前沿观点 / 复盘反思
```

这条线不是附属评论区，而是资产进化的上下文与证据：

- 吐槽记录真实摩擦，但需要区分事实、判断和情绪。
- 讨论补充反例、替代方案和复现结果。
- 实验记录保存输入、输出、日志、截图和版本。
- 技术小结沉淀阶段性方法。
- 深度文章解释长期判断、失败原因和认知变化。

两条线必须能够互相追溯：

- 一个 Resource 可以派生多个 Skill。
- 一个 Skill 可以关联多次 Evaluation、多个版本和多个 Example。
- Quick Take、Evaluation 和 Note 都可以关联同一个 Resource、Skill、版本或实验任务。
- Note 可以引用原始吐槽与评测；Skill 可以引用形成它的 Note 和证据。

## 4. 信息架构

一级导航最终收口为四个稳定入口。

### 4.1 找灵感（灵感池）

统一承载随机发现、候选资源和历史工具：

- 模型
- 软件与编辑器
- API 与在线服务
- GitHub 仓库与开源库
- 网站、应用和 Demo
- 文章、论文、课程和数据集
- 一句话灵感
- 待测清单

评测是灵感池条目的核心能力，不需要再为模型、软件或候选项目建立彼此割裂的产品叙事。

### 4.2 Skill 库

只陈列已经具有一定复用价值的资产：

- Prompt
- Template
- Workflow
- Agent Skill
- Coding / Image / Writing Recipe
- 多个 Skill 组成的技能包

Skill 不等于一段 Prompt。一个成熟 Skill 应逐步具备：

- 目标与适用场景
- 输入要求和输出契约
- 指令、Prompt 或执行步骤
- 依赖的模型、软件和环境
- 示例输入与输出
- 失败模式与适用边界
- 版本与变更记录
- Quick Start
- 评测摘要和最近验证时间

### 4.3 吐槽室

低门槛、异步优先的真实讨论流，支持：

- 吐槽
- 提问
- 求证
- 发现
- 踩坑
- 替代方案

社区原则：

> 允许情绪，但区分事实与判断；吐槽体验，不攻击个人；能说明版本、场景和证据的内容优先展示。

第一阶段复用现有匿名身份、评论、点赞、举报、限流和审核，不急于引入实时聊天室。

### 4.4 洞笔记

承载稍微沉底之后的内容：

- 技术小结
- 正式评测报告
- 前沿观察
- 项目复盘
- 自我反思
- 方法论与 Quick Start 指南

现有 Article / ArticleColumn 继续使用，后续补充文章类型和关联 Subject。

### 4.5 AI 造物台

造物台不是独立内容支柱，而是贯穿四个入口的 **AI 炼成助手**：

- 从 URL 提取资源元数据并查重
- 为不同资源生成评测计划和任务集
- 把原始体验整理成结构化评测草稿
- 从多条吐槽中提取共识、争议和待验证问题
- 把验证结果整理成 Prompt、Workflow、Quick Start 或技能包
- 比较 Skill 版本并生成回归测试
- 将讨论和实验记录沉淀成洞笔记草稿

AI 负责整理、建议和连接，不替代真实测试，不伪造证据。所有 AI 生成或整理内容应标记来源、模型和生成时间，并保留原始输入。

## 5. 统一内容对象与命名

| 名称 | 定义 | 当前对应 |
|---|---|---|
| Inspiration | 原始念头、链接或待验证发现 | Candidate 的早期语义、零散来源 |
| Resource | 模型、软件、服务、仓库、站点、知识资源或项目 | Target、Candidate、部分 App |
| Skill | 从灵感和评测中炼成的可复用资产 | Skill、部分 Target.prompt |
| Evaluation | 带上下文、任务、证据和结论的正式评测 | Review 的升级方向 |
| Quick Take | 吐槽、快评、提问、求证和阶段性发现 | Review/Comment 的新内容类型 |
| Note | 技术小结、评测报告、前沿观点和复盘 | Article |
| Example | Skill 输出案例、Demo 或评测证据 | 部分 App / Gallery |

`Subject` 是技术层面的统一引用概念，不直接作为用户界面名称。

Candidate 应逐步变成 Resource 的成熟度状态，而不是与 Resource 并列的内容类型。

## 6. 评测规范

正式 Evaluation 至少应记录：

- 被测对象与版本
- 使用模型、软件、系统或运行环境
- 实际任务与输入
- 输出、截图、日志、代码或可访问链接
- 重复次数或复现步骤
- 维度评分
- 适用场景
- 失败模式与限制
- 总体结论
- 测试日期

不同对象使用不同模板：

| 对象 | 核心维度 |
|---|---|
| Skill | 指令遵循、输出质量、可复现性、跨模型兼容、成本效率 |
| 模型 | 任务完成率、推理质量、稳定性、速度、成本、可控性 |
| 软件 | 功能价值、稳定性、学习成本、集成能力、价格、隐私 |
| 普通资源 | 权威性、时效性、深度、可用性、复现性、时间价值 |

五星评分可以保留，但不能作为唯一证据。Quick Take 可以不评分，历史自由评测以 `Evaluation v1` 兼容展示。

## 7. 历史数据迁移原则

先统一语义和展示，再逐步迁移物理模型，不进行一次性破坏性重写。

| 现有对象 | 新语义 | 处理原则 |
|---|---|---|
| `Target.editor` / `coding` | 软件或服务 Resource | 保留 ID、slug、评测和旧路由 |
| `Target.model` | 模型 Resource | 直接进入找灵感和资源详情 |
| `Target.prompt` | Skill 或 Prompt Resource | 规则初筛后人工复核 |
| `Candidate` | Resource 成熟度状态 | 晋升时不复制对象、不割裂历史 |
| `App` | Project/Demo Resource 或 Skill Example | 根据关联关系分类 |
| `Skill` | Reusable Artifact | 扩展版本、依赖、Quick Start 和评测协议 |
| `Review` | Evaluation v1 或 Quick Take | 保留原 ID、作者、评分、评论和点赞 |
| `Article` | Note | 增加内容类型和 Subject 关联 |

硬约束：

- 不删除旧数据。
- 不改变历史 Review ID。
- 保留旧 slug 和可访问路径。
- Candidate 进入下一阶段后，原评测、评论和点赞仍在同一详情中可见。
- 新结构化评测使用版本化 schema。
- 新路由稳定后再通过 redirect 和 canonical 收口。

## 8. 首页叙事

推荐首屏：

```text
空心树洞
大浪淘沙，找寻灵感

从一个念头、一条链接或一次踩坑出发，
经过试用、评测与反复打磨，
把值得留下的经验炼成可复用的模板、提示词、工作流和技能包。

[去找灵感] [浏览 Skill] [进入吐槽室]
```

首页内容顺序：

1. 最近炼成的 Skill 与技能包
2. 正在试用和评测的灵感资源
3. 最近验证与淘汰记录
4. 热门吐槽、问题和讨论
5. 最新技术小结与深度文章
6. AI 炼成助手入口

## 9. 产品原则

1. **同一对象持续生长**：状态变化不创建断裂的新对象。
2. **证据优于声量**：热度用于发现，证据决定可信度。
3. **过程和结果同等重要**：失败、吐槽和判断变化必须可追溯。
4. **复用优于收藏**：内容最终应帮助用户快速开始或避免重复踩坑。
5. **兼容优于重写**：历史数据先挂接、再迁移，不为模型整洁牺牲内容资产。
6. **AI 辅助而不代替**：AI 不把未经验证的内容包装成事实。
7. **版本和时效可见**：模型、软件、Skill 和结论都可能过期。

## 10. 衡量方式

北极星指标：

> 每月从灵感池进入 `reusable` 或 `packaged` 状态、且至少包含一次带上下文和证据验证的资产数量。

辅助指标：

- 灵感到首次试用转化率
- 试用到验证转化率
- 验证到可复用转化率
- 含证据评测占比
- 最近 90 天有效验证占比
- 吐槽转正式评测数量
- 评测转技术笔记数量
- Skill 被复制后产生验证反馈的比例

收录量、评论数和五星均分只能作为辅助指标。

## 11. 实施顺序

1. 统一产品文档、品牌表达与命名。
2. 建立 Resource/Skill 的统一展示适配层和成熟度生命线。
3. 将 Candidate 收进 Resource 生命周期，修复晋升后的历史割裂。
4. 区分 Evaluation 与 Quick Take，增加结构化评测协议。
5. 建立吐槽室和洞笔记与 Subject 的关联。
6. 扩展 Skill 的版本、Quick Start、依赖、示例和技能包能力。
7. 接入真实 AI 炼成能力，并保留来源与人工确认。
8. 在展示和迁移稳定后，再评估 Target/App/Candidate 的物理合并。
